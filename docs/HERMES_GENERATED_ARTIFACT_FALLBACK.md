# Hermes Generated Artifact Envelope Fallback

## Why this exists

Isolated staging validation of Chat V2 commit `3d2de7ecbcfb422dbbc12b918175926326550b59` proved that the current Hermes runtime understands image input, but does not expose the custom OpenAI `return_artifact` function tool to the agent. The agent created a backend-local file and returned its path instead.

Chat V2 must never trust a backend filesystem path or remote URL as a user-downloadable file. The fallback therefore transfers generated content inline and stores it again through the Chat Artifact Service.

## Activation

The fallback is disabled by default.

```text
HERMES_PROTOCOL=openai
HERMES_ARTIFACT_TOOL_ENABLED=true
HERMES_ARTIFACT_ENVELOPE_ENABLED=true
```

`HERMES_ARTIFACT_TOOL_ENABLED` retains the preferred OpenAI function-tool path. `HERMES_ARTIFACT_ENVELOPE_ENABLED` adds the fallback contract when the gateway strips or hides that tool.

## Model contract

Chat V2 inserts a system message instructing Hermes to:

1. use `return_artifact` when available;
2. never create or return a backend-local path as the downloadable result;
3. never return a remote URL for Chat V2 to trust;
4. use one inline block per file when the function tool is unavailable.

```text
<CHAT_V2_ARTIFACT>{"filename":"result.txt","mime_type":"text/plain","content_text":"hello"}</CHAT_V2_ARTIFACT>
```

The JSON object may contain only:

- `filename`;
- `mime_type` or `mimeType`;
- exactly one of `content_text`/`contentText` or `content_base64`/`contentBase64`.

Fields such as `path`, `storagePath`, `url`, or any other unrecognized key are rejected.

## Streaming behavior

The parser recognizes markers even when they are split across arbitrary response chunks. Normal assistant text continues to stream with a short marker look-behind buffer. Envelope JSON is never shown in the message body.

Hermes pseudo-content equal to `event: hermes.tool.progress` is treated as internal progress and is not displayed to the user.

## Persistence and security

After parsing, generated bytes still pass through the existing Chat V2 controls:

- inline-content requirement;
- canonical base64 or UTF-8 conversion;
- generated-artifact size limit;
- MIME normalization;
- server-generated storage name;
- Conversation and assistant-message ownership;
- authenticated reload and download.

If both a native/tool artifact and an envelope artifact are returned in the same response, Chat V2 fails explicitly rather than creating duplicates.

## Verification

Required tests cover:

- function-tool regression;
- marker splits across streaming chunks;
- malformed and unfinished markers;
- path and URL rejection;
- duplicate native/envelope rejection;
- API stream to `artifact.created`;
- assistant ownership;
- reload persistence;
- byte-for-byte download;
- desktop, mobile, authentication, container, preflight, and recovery regression.

Real Hermes staging verification requires a separate approval for the new exact SHA. The previous approved staging SHA remains deployed until that approval is granted.
